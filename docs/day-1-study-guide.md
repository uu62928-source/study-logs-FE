# 1日目 学習ガイド

## 今日のゴール

Reactアプリを「画面の集まり」ではなく、責務と依存方向を持つ構造として説明できるようになる。

## 必要な知識

### 1. 境界は変更理由で分ける

- `app`: アプリ全体の起動、ルーティング、Provider、全体スタイル
- `features`: ユーザー価値を持つ機能。UI、状態、ドメイン、API境界を機能単位でまとめる
- `shared`: 複数featureで本当に共有する、業務知識を持たない部品
- `test`: テスト全体の初期設定と共通fixture

ファイルの種類ではなく「何の変更で一緒に変わるか」で近くに置く。最初から共通化せず、再利用の実績ができてから`shared`へ移す。

### 2. 依存方向を一方向にする

```text
app → features → shared
```

`shared`は`features`を知らない。feature同士も直接importせず、必要になったら`app`で組み合わせる。これにより循環依存と、変更の連鎖を抑える。

### 3. feature内部にも責務がある

機能が育ったときだけ、次の小さな境界を追加する。

- `ui`: 表示とユーザー操作
- `model`: UI状態、状態遷移、custom hook
- `domain`: 業務ルールを表す型と純粋関数
- `api`: 通信、DTO、外部データの検証と変換

空の階層を先回りして量産しない。今日の最小画面には`ui`だけがあればよい。

### 4. TypeScriptのstrict設定は仕様漏れを見つける

- `strict`: null、関数引数、推論などを総合的に厳格化する
- `noUncheckedIndexedAccess`: 配列や辞書の参照結果を「存在しないかもしれない」と扱う
- `exactOptionalPropertyTypes`: 「プロパティがない」と「値がundefined」を区別する

厳格さは型エラーを増やすためではなく、実行時に曖昧になる仕様を実装前に発見するために使う。

### 5. importの公開面を小さくする

外部からfeatureを使うときは、そのfeature直下の`index.ts`だけを公開入口にする。ただしbarrel exportを全階層に作ると、依存元が見えにくく循環依存も起きやすい。公開境界にだけ使う。

### 6. テストしやすい構造

UIから計算や業務ルールを純粋関数へ分けると、Reactを起動せず高速に検証できる。外部通信を`api`境界へ閉じ込めると、UIテストで置き換えやすい。テスト容易性は設計の結果であり、後付けの技法ではない。

## 今日、説明できれば合格

1. なぜ機能単位でコードを近くに置くのか
2. `features`と`shared`の違い
3. なぜ依存方向を一方向にするのか
4. 厳格なTypeScript設定が防ぐ問題
5. 今は作らなかった階層と、その理由
